<a id="read-komandi"></a>
## Read команди

### `cat` — преглед на целия файл
Показва цялото съдържание на файл наведнъж. Подходящ за кратки файлове — при дълги е по-удобно да използваш `less`, `head` или `tail`.
```bash
cat file.txt                  # показва целия файл
cat -n file.txt               # с номерирани редове
cat file1.txt file2.txt       # показва двата файла последователно
```

---

### `head` — началото на файл
Показва първите редове на файл. По подразбиране — първите **10 реда**.
```bash
head file.txt                 # първите 10 реда
head -n 20 file.txt           # първите 20 реда
head -n -5 file.txt           # всичко БЕЗ последните 5 реда
head -c 100 file.txt          # първите 100 байта
```

| Флаг | Значение |
|------|----------|
| `-n <N>` | показва първите N реда |
| `-n -<N>` | показва всичко без последните N реда |
| `-c <N>` | показва първите N байта вместо редове |
| `-q` | не показва имената на файловете при множество входа |
| `-v` | винаги показва имената на файловете |

---

### `tail` — краят на файл
Показва последните редове на файл. По подразбиране — последните **10 реда**. Особено полезен за следене на лог файлове в реално време.
```bash
tail file.txt                 # последните 10 реда
tail -n 20 file.txt           # последните 20 реда
tail -n +5 file.txt           # всичко ОТ ред 5 нататък (пропуска първите 4)
tail -c 200 file.txt          # последните 200 байта
tail -f server.log            # следи файла в реално време
tail -f -n 50 server.log      # последните 50 реда + следи в реално време
```

| Флаг | Значение |
|------|----------|
| `-n <N>` | показва последните N реда |
| `-n +<N>` | показва съдържанието започвайки от ред N |
| `-c <N>` | показва последните N байта |
| `-f` | follow — следи файла за нови редове в реално време |
| `-F` | като `-f`, но работи дори ако файлът се презапише/ротира |
| `-q` | не показва имената на файловете |

> **`-f` vs `-F`:** `-f` следи конкретния файл по inode — ако файлът се изтрие и пресъздаде (напр. при лог ротация), `tail -f` спира да работи. `-F` следи по **име** и се reconnectва автоматично — по-надежден за продуктионни логове.

<a id="wc"></a>
## `wc` — броене на редове, думи и символи

`wc` (*word count*) показва колко реда, думи, байтове или символи има във файл или вход от стандартния поток.

### Основен синтаксис

```bash
wc file.txt                  # редове, думи и байтове
wc file1.txt file2.txt       # статистика за всеки файл + общо
cat file.txt | wc            # броене от pipe/stdin
```

---

### Най-често използвани флагове

```bash
wc -l file.txt               # брой редове
wc -w file.txt               # брой думи
wc -c file.txt               # брой байтове
wc -m file.txt               # брой символи
wc -L file.txt               # дължина на най-дългия ред
```

| Флаг | Значение |
|------|----------|
| `-l` | брой редове |
| `-w` | брой думи |
| `-c` | брой байтове |
| `-m` | брой символи |
| `-L` | дължина на най-дългия ред |

---

### Практически примери

```bash
wc -l *.log                  # колко реда има във всеки .log файл
ls -1 | wc -l                # колко файла/реда връща командата
grep -r "TODO" . | wc -l    # колко TODO-та има в проекта
find . -type f | wc -l       # колко файла има общо
```

> **Бележка:** `wc -c` брои байтове, а `wc -m` брои символи. При UTF-8 текст с кирилица резултатът може да е различен.

## inode

Всеки файл в Linux е представен от два отделни обекта: **inode** и **directory entry**.

**inode** (index node) е структура от данни на диска, която пази *всичко за файла* — с изключение на неговото **име**. Всеки inode е идентифициран с уникален номер в рамките на файловата система.

#### Какво съдържа inode-ът:
- Тип на файла (обикновен файл, директория, символна връзка…)
- Права за достъп (`rwxr-xr-x`)
- Собственик (UID) и група (GID)
- Размер на файла
- Timestamps — `atime` (последен достъп), `mtime` (последна промяна на съдържание), `ctime` (последна промяна на метаданни)
- Брой hard links, сочещи към него
- Указатели към блоковете на диска, където е записано съдържанието

**Какво НЕ съдържа:** името на файла (то се пази в директорията) и самото съдържание (inode само сочи към блоковете).

```
Директория:
┌──────────────┬──────────┐
│    Име       │  inode # │
├──────────────┼──────────┤
│  notes.txt   │   42831  │
│  script.sh   │   42832  │
└──────────────┴──────────┘

inode #42831:
┌─────────────────────────┐
│  Тип: обикновен файл    │
│  Права: rw-r--r--       │
│  Собственик: ivan       │
│  Размер: 1024 байта     │
│  Blocks: → [блок 5, 6]  │
└─────────────────────────┘
```

```bash
ls -li file.txt     # показва inode номера в първата колона
stat file.txt       # подробна информация включително inode
df -i               # брой свободни/използвани inodes на дяла
```

> **Интересен факт:** Можеш да запълниш диска не само като изчерпаш пространството, но и като изчерпаш всички inodes. Ако inodes свършат, не можеш да създадеш нов файл — дори да има свободно място на диска.

## symlink

**Symbolic link** (symlink, мек линк) е специален тип файл, чието съдържание е **пътят** до друг файл или директория. Работи като пряк път — когато отвориш symlink, системата автоматично те пренасочва към оригинала.

### Hard link vs Symbolic link

**Hard link** е второ директорийно указание към същия inode. И оригиналът, и линкът сочат към **едни и същи данни** на диска. Изтриването на единия не унищожава файла — данните съществуват докато броят на hard links стане 0.

**Symbolic link** е отделен файл с нов inode, чието съдържание е само пътят до оригинала. Ако оригиналът се изтрие, symlink-ът "виси" и сочи към несъществуващ файл (*dangling symlink*).

```
Hard link:
  fileA  ──┐
            ├──► inode #42831 ──► [данни на диска]
  fileB  ──┘

Symbolic link:
  notes.txt (inode #42831) ──────────────► [данни на диска]
  shortcut  (inode #43001) ──► "/home/ivan/notes.txt"
                                      │
                                      └──► препраща към notes.txt
```

| Свойство | Hard Link | Symbolic Link |
|----------|-----------|---------------|
| Собствен inode | не (споделя) | да |
| Работи между файлови системи | ❌ | ✅ |
| Може да сочи към директория | ❌ | ✅ |
| При изтриване на оригинала | не се влияе | dangling link |
| Вижда се с `ls -l` | изглежда като нормален файл | показва `->` |

### Broken symbolic links

Symbolic линковете могат да бъдат и в **`broken` състояние**. Това се случва когато пътят, записан в symlink-а, е **релативен**, а самият symlink бъде преместен с `mv` на друго място във файловата система.

След преместването релативният път вече не сочи към правилната цел и линкът става **dangling / broken**.

При `ls`:

- нормалните **symlink** файлове се показват в **цианов (светло син)** цвят
- **broken symlink** файловете се показват в **червен** цвят

<a id="ln"></a>
## `ln` — създаване на връзки

`ln` създава hard links и symbolic links. По подразбиране (без флаг) прави **hard link**.

```bash
ln оригинал линк_name           # hard link
ln -s оригинал линк_name        # symbolic link (symlink)
```

### Hard link
```bash
ln notes.txt notes_backup.txt
ls -li notes.txt notes_backup.txt
# и двата имат един и същ inode номер
```

### Symbolic link
```bash
ln -s /home/ivan/projects/web ~/web          # symlink в home към проекта
ln -s /var/log/nginx/access.log ~/logs       # бърз достъп до лог
ln -s ../config/settings.json ./settings     # релативен symlink
```

| Флаг | Значение |
|------|----------|
| `-s` | създава symbolic link вместо hard link |
| `-f` | force — презаписва ако линкът вече съществува |
| `-n` | не следва вече съществуващ symlink директория |
| `-v` | verbose — показва какво е създадено |
| `-r` | използва релативен път при symlink (автоматично изчислява) |

```bash
ln -sf /new/path/file.txt existing_link     # обновява съществуващ symlink
ln -sv source.txt link.txt                  # показва: 'link.txt' -> 'source.txt'

ls -l ~/web
# lrwxrwxrwx 1 ivan ivan 26 Mar 10 14:22 web -> /home/ivan/projects/web
# └─ 'l' в началото = symbolic link

readlink web              # показва накъде сочи symlink-ът
readlink -f web           # показва абсолютния реален път (resolve all)
```

> **Добра практика:** При създаване на symlinks използвай абсолютни пътища, за да избегнеш счупени линкове при преместване.

<a id="find"></a>
## `find` — търсене в файловата система

`find` обхожда директории рекурсивно и търси файлове по различни критерии. Изключително мощна команда — може не само да намира, но и да **изпълнява действия** върху намерените файлове.

### Основен синтаксис

```bash
find [НАЧАЛНА_ДИРЕКТОРИЯ] [ТЕСТОВЕ] [ДЕЙСТВИЯ]

find .                       # всички файлове в текущата директория (рекурсивно)
find /home/ivan              # от конкретна директория
find . -name "*.txt"         # всички .txt файлове
```

---

### Tests — филтри за търсене

Тестовете определят *кои файлове* да бъдат намерени.

#### По име и тип
```bash
find . -name "notes.txt"        # точно съвпадение (case-sensitive)
find . -iname "notes.txt"       # без значение главни/малки букви
find . -name "*.log"            # всички .log файлове
find . -name "file[0-9].txt"    # file0.txt до file9.txt (glob pattern)
find . -type f                  # само обикновени файлове
find . -type d                  # само директории
find . -type l                  # само symbolic links
```

| Стойност на `-type` | Значение |
|---------------------|----------|
| `f` | обикновен файл |
| `d` | директория |
| `l` | symbolic link |
| `b` | block device |
| `c` | character device |
| `p` | named pipe |
| `s` | socket |

#### По размер
```bash
find . -size +100M              # файлове ПО-ГОЛЕМИ от 100MB
find . -size -1k                # файлове ПО-МАЛКИ от 1KB
find . -size 50c                # файлове ТОЧНО 50 байта
find . -empty                   # празни файлове и директории
```

| Суфикс | Единица |
|--------|---------|
| `c` | байтове |
| `k` | килобайти (1024) |
| `M` | мегабайти |
| `G` | гигабайти |

#### По дата
```bash
find . -mtime -7                # променени през последните 7 дни
find . -mtime +30               # не са променяни от повече от 30 дни
find . -mtime 0                 # променени днес (последните 24ч)
find . -newer reference.txt     # по-нови от reference.txt
find . -mmin -60                # променени преди по-малко от 60 минути
```

| Флаг | Timestamp |
|------|-----------|
| `-mtime` | последна промяна на **съдържанието** |
| `-atime` | последен **достъп** |
| `-ctime` | последна промяна на **метаданните** (chmod, chown…) |
| `-mmin` | като `-mtime`, но в **минути** |

#### По права и собственик
```bash
find . -perm 644                # точно тези права
find . -perm -u+x               # файлове, изпълними от собственика
find . -perm /o+w               # файлове, записваеми от всички (опасно)
find . -user ivan               # файлове на потребител ivan
find . -group developers        # файлове на група developers
find . -nouser                  # файлове без съществуващ собственик
```

#### Логически оператори
```bash
find . -name "*.txt" -and -size +1M        # И двете условия (по подразбиране)
find . -name "*.log" -or -name "*.tmp"     # ПОНЕ едно условие
find . -not -name "*.git"                  # НЕ отговаря на условието
find . \( -name "*.jpg" -or -name "*.png" \) -size +5M
```

---

### Global Options — глобални опции

Засягат цялото поведение на `find`. По добра практика се поставят **преди тестовете**.

```bash
find . -maxdepth 1 -name "*.txt"           # само в текущата директория (без поддиректории)
find . -maxdepth 2 -type d                 # директории до 2 нива надолу
find . -mindepth 2 -name "*.log"           # само в поддиректории (пропуска корена)
find . -mindepth 1 -maxdepth 3 -type f     # между 1 и 3 нива дълбочина
```

| Опция | Значение |
|-------|----------|
| `-maxdepth <N>` | не влиза по-дълбоко от N нива |
| `-mindepth <N>` | пропуска файлове по-плитки от N нива |
| `-depth` | обработва съдържанието на директория преди самата нея |
| `-follow` | следва symbolic links |
| `-xdev` | не преминава в друга файлова система |

### Actions — действия върху намерените файлове

#### `-print` и `-printf`
`-print` е действието по подразбиране — отпечатва пътя на всеки намерен файл. `-printf` дава пълен контрол върху формата на изхода.

```bash
find . -name "*.log" -print              # стандартен изход (и без -print)
find . -type f -printf "%p\n"            # само пътя
find . -type f -printf "%f\t%s\n"        # само името и размера (tab разделени)
find . -type f -printf "%p %u %m\n"      # път, собственик, права в octal
find . -type f -printf "%-40p %8s\n"     # наредено в колони
```

| Placeholder | Значение |
|-------------|----------|
| `%p` | пълен път до файла |
| `%f` | само името на файла (без пътя) |
| `%s` | размер в байтове |
| `%u` | собственик (username) |
| `%g` | група |
| `%m` | права в octal (напр. `644`) |
| `%M` | права в символен формат (напр. `-rw-r--r--`) |
| `%t` | дата на последна промяна |
| `%i` | inode номер |
| `%l` | накъде сочи symlink (само за links) |
| `\n` | нов ред |
| `\t` | табулация |

#### `-exec` — изпълни команда върху намерените файлове

`-exec` изпълнява произволна команда за всеки намерен файл. **`{}`** е placeholder — на негово място `find` поставя пътя на текущия файл. Командата **задължително** завършва с `\;` или `+`.

```bash
find . -name "*.log" -exec cat {} \;          # показва съдържанието на всеки .log
find . -type f -exec chmod 644 {} \;          # задава права на всички файлове
find . -name "*.tmp" -exec rm {} \;           # изтрива .tmp файлове
find . -type f -exec grep -l "TODO" {} \;     # файлове съдържащи "TODO"
```

**`\;` vs `+`** — важна разлика в производителността:

```bash
find . -name "*.txt" -exec ls -l {} \;
# Изпълнява: ls -l file1.txt
#            ls -l file2.txt      ← отделна команда за ВСЕКИ файл

find . -name "*.txt" -exec ls -l {} +
# Изпълнява: ls -l file1.txt file2.txt file3.txt  ← всички НАВЕДНЪЖ
```

`+` е по-бърз защото стартира значително по-малко процеси. Ползвай `\;` само когато командата трябва да работи поотделно за всеки файл.

```bash
find /var/log -name "*.log" -mtime +30 -exec gzip {} \;      # компресира стари логове
find . -newer reference.txt -exec cp {} /backup/ \;           # копира новите файлове
find . -type f -exec file {} \; | grep "PNG image"            # намира PNG по съдържание
```

#### `-delete`

`-delete` е вграденото действие за изтриване — по-бързо от `-exec rm`, защото не стартира нов процес.

```bash
find . -name "*.tmp" -delete                  # изтрива всички .tmp файлове
find . -type f -empty -delete                 # изтрива всички празни файлове
find . -name "*.log" -mtime +90 -delete       # изтрива логове по-стари от 90 дни
```

> ⚠️ **Внимание с `-delete`:** Винаги тествай командата **без** `-delete` първо, за да видиш кои файлове ще бъдат засегнати. Добавяй `-delete` едва след като си сигурен.

```bash
find . -name "*.tmp" -mtime +7           # 1. виж кои файлове ще засегнеш
find . -name "*.tmp" -mtime +7 -delete   # 2. изтрий ги
```

---

### Практически примери

```bash
# Намери всички файлове по-големи от 100MB и покажи размерите им
find / -type f -size +100M -printf "%s\t%p\n" 2>/dev/null

# Намери и изтрий всички __pycache__ директории в проект
find . -type d -name "__pycache__" -exec rm -rf {} +

# Намери файлове с опасни права (записваеми от всички)
find /etc -perm /o+w -type f

# Намери всички symlinks и покажи накъде сочат
find . -type l -printf "%p -> %l\n"

# Намери дублирани конфигурационни файлове
find /etc -name "*.conf" -maxdepth 2 -printf "%f\t%p\n" | sort
```

<a id="pipes"></a>
## Pipes

`|` (pipe) свързва изхода на една команда с входа на друга. Вместо резултатът да се показва директно в терминала, той се подава към следващата команда за допълнителна обработка.

```bash
cat file.txt | wc -l
# брои редовете във файла

ls -l | grep ".txt"
# показва само .txt файловете

dmesg | less
# преглежда дълъг изход страница по страница
```

Pipes позволяват комбиниране на малки команди в по‑мощни вериги за обработка на текст и файлове.

<a id="race-condition"></a>

## Race condition и mutual exclusion

В съвременна ОС решенията кой процес да използва процесора се взимат от самата ОС. **Невъзможно е да се детерминира предварително кога работата на даден процес ще прекъсне** и управлението ще бъде предадено на друг.

Когато два или повече процеса достъпват **общ ресурс** без синхронизация, се случва **race condition (състезание за ресурс)** — резултатът зависи от случайния ред на изпълнение.

```
Процес A чете count = 5
Процес B чете count = 5      ← и двата виждат 5
Процес A записва count = 6
Процес B записва count = 6   ← очакваме 7, получаваме 6!
```

**Решението** е **mutual exclusion** — забрана ресурсът да се използва от повече от един процес едновременно.

Сегментите от код, които могат да бъдат достъпени от няколко процеса едновременно, се наричат **критични секции**. Ако те станат **атомарни** (неприекъсваеми), проблемът се решава.

<a id="spinlock"></a>

## Spinlock

Най-простият механизъм — защита с **един бит (lock)**:
- `lock = 0` → ресурсът е свободен
- `lock = 1` → ресурсът е зает

```
spin_lock(lock):
    while lock == 1:
        ; // чакай (въртиш се на празно)
    lock = 1

spin_unlock(lock):
    lock = 0
```

Всяка критична секция трябва да започва с `spin_lock()` и да завършва с `spin_unlock()`.

### Проблеми при spinlock

| Проблем | Описание |
| --- | --- |
| **Вложен достъп** | Ако в критичната секция се стартира подпроцес, опитал се да достъпи същия lock, ще настъпи безкраен цикъл |
| **Хардуерно прекъсване** | Периферно устройство, опитало се да достъпи секцията, ще зацикли |
| **Timer interrupt** | ОС използва периодичен timer за превключване на процеси. Ако текущият процес е в критична секция и бъде прекъснат, всички останали процеси, искащи да влязат в секцията, ще зациклят — губи се ефективност |

**Частично решение** — забрана на прекъсванията около критичната секция:

```
disable_interrupts()
spin_lock(lock)
    // критична секция
spin_unlock(lock)
enable_interrupts()
```

Но ако **два процеса едновременно** забранят прекъсванията и единият зацикли, другият не може да бъде прекъснат за нищо друго — неприемливо за ефективността.

<a id="semaphore"></a>

## Семафор — структура и операции

Приспиването на процес изисква запазване на всичките му данни — т.нар. **смяна на контекста (context switching)**. Семафорът е **механизъм за синхронизация от високо ниво**, който използва именно това.

### Структура на семафор

```c
struct Semaphore {
    int cnt;    // брой процеси, допустими до ресурса
    Queue Q;    // опашка от чакащи процеси (пазим техните PID-ове)
    int lock;   // spinlock за защита на самия семафор
};
```

### Операции

**`init(int n)`** — инициализация:
```
cnt = n
Q = празна опашка
lock = 0
```

**`wait()`** — опит за достъп до ресурса:
```
spin_lock(lock)
cnt = cnt - 1
if cnt < 0:
    добави текущия PID в Q
    spin_unlock(lock)
    block()          // приспи процеса (ядрото го изважда от scheduler-а)
else:
    spin_unlock(lock)
```

**`signal()`** — освобождаване на ресурса:
```
spin_lock(lock)
cnt = cnt + 1
if Q не е празна:
    pid = Q.dequeue()
    spin_unlock(lock)
    wakeup(pid)      // събуди процеса (ядрото го връща в scheduler-а)
else:
    spin_unlock(lock)
```

> **Забележка:** `spin_unlock()` при `wait()` се поставя **преди** `block()` — ако не освободим lock-а преди да заспим, той остава заключен завинаги.
> При `signal()`, `spin_unlock()` се поставя **преди** `wakeup()` — `wakeup()` може да отнеме значително време.

### Силен vs слаб семафор

| Тип | Опашка Q | Гаранция |
| --- | --- | --- |
| **Силен** | FIFO опашка | Процесите се събуждат в реда на изчакване — **без starvation** |
| **Слаб** | Произволна структура | Не е гарантирано кой процес ще се събуди |

### Как `cnt` кодира информация

| Стойност на `cnt` | Значение |
| --- | --- |
| `cnt > 0` | Толкова процеса могат **още да влязат** без блокиране |
| `cnt = 0` | Ресурсът е точно зает — следващият ще блокира |
| `cnt < 0` | `\|cnt\|` процеса **чакат** в опашката |

<a id="deadlock"></a>

## Deadlock

**Deadlock** е ситуация, при която два или повече процеса чакат безкрайно един друг да освободи ресурс.

### Четирите условия за deadlock

За да възникне deadlock, трябва **всичките четири** условия да са изпълнени едновременно:

| Условие | Описание |
| --- | --- |
| **Mutual exclusion** | Ресурсът може да се ползва само от 1 процес едновременно |
| **Hold & wait** | Процес притежава ресурс и чака за друг |
| **No preemption** | Ресурс не може да се отнеме принудително — процесът трябва сам да го освободи |
| **Цикличност** | Съществува цикъл: P1 чака P2, P2 чака P3, ..., Pn чака P1 |

### Стратегии за избягване

**1. Номериране на ресурсите** — процесите заявяват ресурси само в нарастващ ред на номерата. Тъй като не може да се образува цикъл при строго нарастващ ред, deadlock е невъзможен.

**2. Proxy процес (spooler)** — специален процес поема ексклузивния достъп до ресурса. Останалите не достъпват ресурса директно, а го правят чрез proxy-то. Пример: **print spooler** — програмите не пишат директно на принтера, а изпращат заявка към spooler процеса.

## Synchronization patterns

---

<a id="rendezvous"></a>

### Rendezvous (среща)

**Проблем:** Имаме два процеса A и B. Искаме `a1` да се изпълни преди `b2` и `b1` да се изпълни преди `a2` — двата процеса трябва да се "срещнат" преди да продължат.

```
Процес A:        Процес B:
    a1               b1
    // точка на среща
    a2               b2
```

**Решение:**

```
aArrived.init(0)
bArrived.init(0)

Процес A:            Процес B:
    a1                   b1
    aArrived.signal()    bArrived.signal()
    bArrived.wait()      aArrived.wait()
    a2                   b2
```

> **Внимание:** Ако разменим `wait()` и `signal()` в един процес, губим производителност. Ако го направим и в двата — deadlock (и двата чакат, никой не сигнализира).

---

<a id="mutex"></a>

### Mutex (mutual exclusion)

**Проблем:** Два процеса споделят променлива `count`. Искаме достъпът до нея да е ексклузивен.

**Решение:**

```
mutex.init(1)    // 1 = свободен, 0 = зает

Процес A:            Процес B:
    mutex.wait()         mutex.wait()
    count++              count++
    mutex.signal()       mutex.signal()
```

Семафор инициализиран с `1`, използван като mutex, се нарича **бинарен семафор**.

---

<a id="multiplex"></a>

### Multiplex

**Проблем:** Обобщение на mutex — не повече от `n` нишки могат да достъпват критичната секция едновременно.

**Решение:**

```
multiplex.init(n)    // n нишки могат едновременно

Всяка нишка:
    multiplex.wait()
        // критична секция
    multiplex.signal()
```

При `n=1` това е еквивалентно на mutex.

---

<a id="barrier"></a>

### Бариера

**Проблем:** Обобщение на rendezvous за `n` нишки. Първите `n-1` нишки трябва да блокират, докато не пристигне и последната — тогава всички продължават.

**Решение:**

```
// Споделени променливи:
n         = общ брой нишки
count     = 0       // брой пристигнали нишки
mutex.init(1)
barrier.init(0)     // заключена в началото

Всяка нишка:
    mutex.wait()
        count++
        if count == n:
            barrier.signal()    // последната отключва бариерата
    mutex.signal()

    barrier.wait()              // всички чакат тук
    barrier.signal()            // всяка нишка предава нататък ("turnstile" pattern)
```

`barrier.signal()` след `barrier.wait()` е т.нар. **turnstile** — всяка нишка отваря вратата за следващата и така всички n нишки преминават.

---

<a id="turnstile"></a>

### Turnstile (турникет)

**Turnstile** е прост pattern, кръстен на турникетите на метрото — пускат по един наведнъж, но след като първият мине, може да го предаде на следващия.

Реализира се с един семафор инициализиран с `0`:

```
turnstile.init(0)

// Заключване — блокира всички нишки
turnstile.wait()
turnstile.signal()   // предава нататък веднага след преминаване
```

#### Как работи

Семафорът започва от `0`, така че първата нишка, извикала `wait()`, блокира. Когато външен код извика `signal()` (напр. последната пристигнала нишка в бариера), едната чакаща нишка се освобождава. Тя веднага извиква `signal()` отново — освобождава следващата. Така турникетът "пропуска" нишките по верига, по една.

```
Начало:  turnstile = 0   (всички блокират при wait())

                     [нишка 1] → wait() → блокира
                     [нишка 2] → wait() → блокира
                     [нишка 3] → wait() → блокира

Някой извиква signal():
    turnstile = 1
    [нишка 1] се освобождава → минава → signal() → turnstile = 1
    [нишка 2] се освобождава → минава → signal() → turnstile = 1
    [нишка 3] се освобождава → минава → signal() → turnstile = 1
```

#### Употреба в Бариера

Turnstile решава проблем, присъщ на бариерата — **повторна употреба**. Ако използваме само `barrier.signal()` за последната нишка, тя освобождава само едната следваща. Turnstile pattern-ът гарантира, че всичките `n` нишки преминават:

```
// Фаза 1 — изчакай всички да пристигнат
mutex.wait()
    count++
    if count == n:
        turnstile.signal()   // последната отключва турникета
mutex.signal()

// Фаза 2 — всички преминават, всяка предава нататък
turnstile.wait()
turnstile.signal()
```

#### Double Turnstile — за многократна бариера

Проблем: ако бариерата се използва в цикъл, бърза нишка може да мине втория кръг и да достигне бариерата отново, преди бавна нишка да е напуснала първия. Решението е **два turnstile-а**, използвани последователно:

```
// Споделени:
count = 0
mutex.init(1)
t1.init(0)   // turnstile 1 — за влизане
t2.init(1)   // turnstile 2 — за излизане (отключен!)

Всяка нишка (в цикъл):
    // --- Фаза на влизане ---
    mutex.wait()
        count++
        if count == n:
            t2.wait()      // заключи изхода
            t1.signal()    // отключи входа
    mutex.signal()
    t1.wait()              // изчакай всички да влязат
    t1.signal()

    // критична секция

    // --- Фаза на излизане ---
    mutex.wait()
        count--
        if count == 0:
            t1.wait()      // заключи входа
            t2.signal()    // отключи изхода
    mutex.signal()
    t2.wait()              // изчакай всички да излязат
    t2.signal()
```

По всяко време единият turnstile е заключен, а другият — отключен. Това гарантира, че нишките са или **всички вътре**, или **всички вън** — никога смесено.

#### Turnstile в Readers-Writers

В Readers-Writers2 turnstile-ът се използва по различен начин — не за пропускане на всички, а за **временно спиране на читателите** докато писателят изчаква:

```
turnstile.init(1)   // ← инициализиран с 1, не с 0!

Reader:
    turnstile.wait()    // минава ако писател не чака
    turnstile.signal()  // веднага освобождава
    // ... чете

Writer:
    turnstile.wait()    // заключва — нови читатели спират тук
    roomEmpty.wait()    // чака текущите читатели да излязат
    // ... пише
    roomEmpty.signal()
    turnstile.signal()  // отключва — читателите продължават
```

Тук `init(1)` означава, че в нормален режим турникетът е **отворен** — читателите минават без блокиране. Само когато писател чака (`turnstile.wait()` без последващ `signal()`), вратата се заключва.

#### Обобщение

| Употреба | Init | Поведение |
| --- | --- | --- |
| **Бариера** | `0` | Блокира всички докато последната нишка не отключи |
| **Double turnstile** | `0` и `1` | Гарантира, че всички са или вътре, или вън |
| **Readers-Writers** | `1` | Отворен по подразбиране, писателят го заключва временно |

<a id="producer-consumer"></a>

### Producer-Consumer

**Проблем:** Имаме два вида нишки:
- **Producer** — създава обекти и ги добавя в споделен буфер
- **Consumer** — взима обекти от буфера и ги обработва

Синхронизационни ограничения:
- Достъпът до буфера трябва да е ексклузивен (mutex)
- Ако буферът е **празен**, consumer блокира докато producer не добави обект

**Решение:**

```
// Споделени променливи:
mutex.init(1)
items.init(0)    // брой обекти в буфера (в началото 0)
buffer = []

Producer:                    Consumer:
    // създай обект              items.wait()     // чакай докато има обект
    mutex.wait()                 mutex.wait()
        buffer.add(item)             item = buffer.remove()
    mutex.signal()               mutex.signal()
    items.signal()               // обработи item
```

> **Забележка:** При Producer редът на последните два реда може да се размени (`items.signal()` преди `mutex.signal()`). Вариантът показан горе дава малък performance boost — не будим consumer преди да сме освободили mutex-а.

<a id="producer-consumer2"></a>

### Producer-Consumer с краен буфер

**Проблем:** Буферът има краен размер `capacity`. Добавено ограничение:
- Ако буферът е **пълен**, producer блокира докато consumer не вземе обект

**Решение:**

```
// Споделени променливи:
mutex.init(1)
items.init(0)           // брой обекти (consumer чака при 0)
spaces.init(capacity)   // свободни места (producer чака при 0)
buffer = []

Producer:                    Consumer:
    spaces.wait()                items.wait()
    mutex.wait()                 mutex.wait()
        buffer.add(item)             item = buffer.remove()
    mutex.signal()               mutex.signal()
    items.signal()               spaces.signal()
                                 // обработи item
```

`items + spaces = capacity` е инварiantът на системата.

<a id="readers-writers"></a>

### Readers-Writers

**Проблем:** Имаме читатели и писатели, споделящи ресурс. Ограничения:
- Произволен брой **читатели** могат да са в критичната секция едновременно
- **Писателите** трябва да имат ексклузивен достъп
- Докато има писател — никой друг не може да достъпи ресурса

**Решение:**

```
// Споделени променливи:
roomEmpty.init(1)    // 1 = никой в стаята, 0 = има някой
mutex.init(1)        // защита на readers_count
readers_count = 0

// LightSwitch pattern: първият читател заключва, последният отключва
Reader:
    mutex.wait()
        readers_count++
        if readers_count == 1:
            roomEmpty.wait()    // първият читател заключва за писатели
    mutex.signal()

    // четене...

    mutex.wait()
        readers_count--
        if readers_count == 0:
            roomEmpty.signal()  // последният читател освобождава
    mutex.signal()

Writer:
    roomEmpty.wait()    // изчака всички читатели/писатели да излязат
    // писане...
    roomEmpty.signal()
```

Моделът **"първият влизащ заключва, последният излизащ отключва"** се нарича **LightSwitch**.

> **Проблем:** Имаме **starvation на писателите** — ако читателите идват непрекъснато, писателят може да чака безкрайно.

---

<a id="readers-writers2"></a>

### Readers-Writers без starvation на писателите

**Решение:** Добавяме `turnstile` семафор. Когато писател пристигне, той заключва turnstile-а — новите читатели спират да влизат. Писателят изчаква вече влезлите читатели да излязат, след което пише.

```
// Добавяме:
turnstile.init(1)

Reader:
    turnstile.wait()        // ← ново: читателят минава само ако e свободен
    turnstile.signal()      // ← веднага го освобождава (само "проверка")
    mutex.wait()
        readers_count++
        if readers_count == 1:
            roomEmpty.wait()
    mutex.signal()
    // четене...
    mutex.wait()
        readers_count--
        if readers_count == 0:
            roomEmpty.signal()
    mutex.signal()

Writer:
    turnstile.wait()        // ← заключва turnstile: нови читатели спират
    roomEmpty.wait()        // чака текущите читатели да излязат
    // писане...
    roomEmpty.signal()
    turnstile.signal()      // ← отключва turnstile след писане
```

<a id="dining-philosophers"></a>

### Dining Philosophers (Философи на вечеря)

**Проблем:** 5 философа седят около кръгла маса с 5 вилици (по 1 между всеки двама). Всеки философ изпълнява цикъл:

```
while true:
    think()
    get_forks()
    eat()
    put_forks()
```

Философ `i` има вилица `i` вдясно и вилица `(i+1) % 5` вляво.

**Ограничения:**
- Само 1 философ може да притежава конкретна вилица едновременно
- Deadlock невъзможен
- Starvation невъзможна
- Повече от 1 философ може да яде едновременно

**Помощни функции:**
```c
int right_fork(int i) { return i; }
int left_fork(int i)  { return (i + 1) % 5; }
```

**Грешно решение (deadlock!):**
```
forks[5] — всеки инициализиран с 1

get_forks(i):
    forks[right_fork(i)].wait()   // всеки взима дясната вилица...
    forks[left_fork(i)].wait()    // ...после чака лявата → DEADLOCK

put_forks(i):
    forks[right_fork(i)].signal()
    forks[left_fork(i)].signal()
```

Ако всички 5 философа едновременно вземат дясната си вилица — никой не може да вземе лявата → deadlock.

**Правилно решение — Multiplex:**

```
forks[5]       — всеки инициализиран с 1
footman.init(4) — максимум 4 философа на масата едновременно

get_forks(i):
    footman.wait()                  // допускаме само 4 от 5
    forks[right_fork(i)].wait()
    forks[left_fork(i)].wait()

put_forks(i):
    forks[right_fork(i)].signal()
    forks[left_fork(i)].signal()
    footman.signal()
```

С максимум 4 философа на масата, в най-лошия случай всеки взима дясната вилица. Тогава ще остане вилица с двама съседа, всеки с по 1 вилица — единият от тях ще вземе свободната и ще яде. **Deadlock е невъзможен.**

**Гаранция срещу starvation:** Ако си блокиран за дясната вилица, рано или късно десният ти съсед ще я освободи и ти ще си единственият, чакащ за нея.

> **Алтернативно решение:** Поне един философ да е **левичар** (взима лява преди дясна). Ако имаме поне 1 левичар сред десничари, цикъл не може да се образува, следователно deadlock е невъзможен.
>
> **Доказателство:** Deadlock изисква всеки философ да държи 1 вилица и да чака за другата. Ако има левичар — той чака лявата, но неговият ляв съсед (десничар) чака дясната, т.е. чака вилицата на левичара — цикълът се скъсва.

## Обобщение на patterns

| Pattern | Семафори | Идея |
| --- | --- | --- |
| **Rendezvous** | 2 × `init(0)` | Двата процеса се чакат взаимно преди да продължат |
| **Mutex** | 1 × `init(1)` | Само 1 процес в критичната секция |
| **Multiplex** | 1 × `init(n)` | Максимум n процеса едновременно |
| **Бариера** | mutex + barrier | Всички n нишки чакат последната |
| **Producer-Consumer** | mutex + items | Consumer чака при празен буфер |
| **Producer-Consumer2** | mutex + items + spaces | Producer чака при пълен буфер |
| **Readers-Writers** | roomEmpty + mutex | Много четатели или 1 писател |
| **Readers-Writers2** | + turnstile | Избягва starvation на писателите |
| **Dining Philosophers** | forks[] + footman | Multiplex с максимум n-1 участници |

## FD таблицата в ядрото

### Три нива на абстракция

File descriptor-ите не са просто числа — зад тях стоят три нива на структури в ядрото:

```
Ниво 1: FD таблица на процеса (per-process)
  Индекс (FD)  Флагове    Указател към →
  ┌─────┬──────────────┬────────────┐
  │  0  │ O_RDONLY     │ ──────────►│──┐
  │  1  │ O_WRONLY     │ ──────────►│──┼──┐
  │  2  │ O_WRONLY     │ ──────────►│──┼──┤
  │  3  │ O_RDWR       │ ──────────►│──┼──┼──┐
  └─────┴──────────────┴────────────┘  │  │  │  │
                                        │  │  │  │
Ниво 2: Open File Table (системна, споделена)      │
  ┌──────────────────────────────────────┐         │
  │ offset: 0      flags: O_RDONLY  ◄────┘         │
  │ offset: 142    flags: O_WRONLY  ◄───────────────┘
  │ offset: 0      flags: O_WRONLY  ◄──────────┘
  │ ref_count: 2                                    ↑ двата FD сочат
  └──────────────────────────────────────┘            към едно и също
                        │                             open file entry
                        ▼                             (напр. след dup())
Ниво 3: vnode / inode (filesystem)
  ┌──────────────────────────────────────┐
  │ /home/ivan/notes.txt                 │
  │ inode: права, собственик, размер     │
  │ блокове на диска                     │
  └──────────────────────────────────────┘
```

**FD таблицата** е масив, частен за всеки процес. Индексът е FD числото. Всеки запис пази флагове (напр. `O_CLOEXEC`) и указател към open file entry.

**Open File Table** е споделена в цялата система. Пази текущия офсет на файла и reference count — колко FD сочат към нея. Когато два FD сочат към един запис (след `dup()` или `fork()`), те споделят офсета.

**vnode/inode** е представянето на файла на ниво файлова система — права, собственик, данни на диска.

### Стандартните FD при старт

Всеки процес се ражда с три предварително отворени FD:

```
FD 0  →  stdin   →  четене от клавиатурата (или pipe/файл при пренасочване)
FD 1  →  stdout  →  писане в терминала    (или pipe/файл при пренасочване)
FD 2  →  stderr  →  писане в терминала    (за грешки, не се redirect-ва с >)
```

Следващите отворени файлове получават FD 3, 4, 5… — **ядрото избира най-малкия свободен индекс**.

```c
int fd3 = open("a.txt", O_RDONLY);  // получава FD 3
int fd4 = open("b.txt", O_RDONLY);  // получава FD 4
close(fd3);                          // освобождава FD 3
int fd3b = open("c.txt", O_RDONLY); // отново получава FD 3!
```

## Pipes — неименувани канали

### Какво е pipe

Pipe-ът е едноплосочен комуникационен канал в паметта — данните влизат от единия край и излизат от другия. Той е **буфер в ядрото** (обикновено 64KB) без файл на диска.

```
           write end                    read end
Процес A ────────────► [ буфер в ядрото ] ────────────► Процес B
           FD[1]          (pipe buffer)       FD[0]
           (писане)                           (четене)
```

Pipe-ът е **еднопосочен** — данните текат само в едната посока. За двупосочна комуникация са нужни два pipe-а.

### `pipe()` — създаване

```c
#include <unistd.h>

int pipe(int pipefd[2]);
// Попълва масива pipefd с два нови file descriptor-а:
//   pipefd[0] → read end  (четене от pipe-а)
//   pipefd[1] → write end (писане в pipe-а)
// Връща: 0 при успех, -1 при грешка
```

```c
int pfd[2];
if (pipe(pfd) == -1)
    err(1, "pipe");

// pfd[0] → четене (read end)
// pfd[1] → писане (write end)

// Напиши в pipe-а
write(pfd[1], "hello", 5);

// Прочети от pipe-а
char buf[10];
ssize_t n = read(pfd[0], buf, sizeof(buf));
// n == 5, buf == "hello"
```

### Диаграма на pipe между два процеса

```
         fork()
            │
    ┌───────┴───────┐
    │               │
 РОДИТЕЛ          ДЕТЕ
                    │
                    │ exec("ls")
                    ▼
                   ls

До fork() — двата FD са в родителя:
  pfd[0] ──► [pipe буфер] ◄── pfd[1]

След fork() — всеки процес има копие и на двата FD:
  РОДИТЕЛ                    ДЕТЕ (ls)
  pfd[0] ──► [         ] ◄── pfd[1]
  pfd[1] ──► [ буфер   ] ◄── pfd[0]
  (четене)   [         ]     (писане)

Родителят чете (pfd[0]), детето пише (pfd[1]):
  1. Дете затваря pfd[0] (не му трябва четене)
  2. Родителят затваря pfd[1] (не му трябва писане)
  3. Дете пише в pfd[1] → данните отиват в буфера
  4. Родителят чете от pfd[0] ← данните идват от буфера
```

```c
int pfd[2];
pipe(pfd);

pid_t pid = fork();
if (pid == 0) {
    // ── ДЕТЕ: пише в pipe-а ──
    close(pfd[0]);                    // затвори четящия край — не го ползва
    dup2(pfd[1], STDOUT_FILENO);      // stdout → pipe write end
    close(pfd[1]);                    // оригиналният pfd[1] вече не е нужен
    execlp("ls", "ls", NULL);
    err(1, "execlp");
}

// ── РОДИТЕЛ: чете от pipe-а ──
close(pfd[1]);                        // затвори пишещия край — не го ползва

char buf[256];
ssize_t n;
while ((n = read(pfd[0], buf, sizeof(buf))) > 0)
    write(STDOUT_FILENO, buf, n);

close(pfd[0]);
waitpid(pid, NULL, 0);
```

<a id="blocking"></a>

## Как `read()` и `write()` спят

Pipe-ът е буфер с краен размер (~64KB на Linux). Когато той е пълен или празен, системните извиквания **блокират** — процесът спи докато ситуацията не се промени.

### `read()` блокира при празен pipe

```
Pipe буфер: [ празен ]

read(pfd[0], buf, 100) →  процесът спи...
                           ядрото го слага в wait queue

Когато друг процес напише данни:
write(pfd[1], "data", 4) → ядрото събужда четящия процес
                            read() се връща с n=4
```

### `write()` блокира при пълен pipe

```
Pipe буфер: [ ████████████ пълен ████████████ ]  (64KB)

write(pfd[1], big_data, 100000) →  процесът спи...
                                    чака четящият да освободи място

Когато четящият чете:
read(pfd[0], buf, 4096) → освобождава място
                           write() се събужда и продължава
```

### EOF — кога `read()` връща 0

`read()` на pipe връща **0 (EOF) само когато всички write end-ове са затворени**. Докато поне един write end е отворен — `read()` ще блокира и чака, дори буферът да е празен.

```
Ситуация:
  pfd[1] е отворен в родителя И в детето

  Детето приключи и затвори своя pfd[1]
  Но родителят все още държи pfd[1] отворен!
  → read(pfd[0]) ще блокира завинаги         ← DEADLOCK
  → никой вече няма да пише, но pipe-ът
    не знае това, защото write end е отворен

Решение:
  Родителят трябва да затвори своя pfd[1]
  → тогава read(pfd[0]) ще върне 0 когато
    буферът се изпразни                       ← правилно EOF
```

<a id="close-fds"></a>

## Защо затваряме неизползваните FD

### Правилото

След `fork()` и **преди** `exec()`, всеки процес трябва да затвори FD-тата, които няма да използва. Има две причини: ресурси и коректност.

### Причина 1: Deadlock при pipe

Това е най-важната причина. Ако не затвориш неизползваните write end-ове, `read()` никога няма да получи EOF:

```c
int pfd[2];
pipe(pfd);
pid_t pid = fork();

// ГРЕШНО — без close на неизползваните FD:
if (pid == 0) {
    // Детето: иска да пише, но НЕ затваря pfd[0]
    write(pfd[1], "hello", 5);
    _exit(0);
    // pfd[0] и pfd[1] се затварят при _exit()
}

// Родителят: иска да чете, но НЕ затваря pfd[1]
char buf[100];
read(pfd[0], buf, 100);  // ← БЛОКИРА ЗАВИНАГИ!
// Защо? Защото родителят сам държи pfd[1] отворен.
// Ядрото вижда: write end е отворен → не изпраща EOF
// Дори детето да е приключило, родителският pfd[1] е жив.
```

```c
// ПРАВИЛНО:
if (pid == 0) {
    close(pfd[0]);              // детето не чете — затвори read end
    write(pfd[1], "hello", 5);
    close(pfd[1]);
    _exit(0);
}

close(pfd[1]);                  // родителят не пише — затвори write end
char buf[100];
ssize_t n = read(pfd[0], buf, 100);  // ← получава "hello", после EOF
close(pfd[0]);
```

### Причина 2: Изтичане на ресурси

Всеки процес има лимит на отворени FD (~1024 по подразбиране). Незатворените FD заемат място — при дълги програми, сървъри или цикли от `fork()` това води до "too many open files" грешка.

### Причина 3: Exec наследява FD

`exec()` по подразбиране наследява всички отворени FD. Новата програма може неволно да чете/пише в pipe, файл или сокет, за който нищо не знае. Флагът `O_CLOEXEC` при `open()` или `fcntl()` автоматично затваря FD при `exec()`:

```c
int fd = open("secret.txt", O_RDONLY | O_CLOEXEC);
// fd се затваря автоматично при exec() — новата програма не го вижда
```

## Наследяване на pipe FD след `fork()`

### Какво се случва при fork()

`fork()` **копира FD таблицата** на родителя в детето. Всеки отворен FD — включително pipe FD — е достъпен и в двата процеса след fork().

```
Преди fork():
  Родител:  FD 0(stdin), FD 1(stdout), FD 2(stderr), FD 3(pfd[0]), FD 4(pfd[1])

След fork():
  Родител:  FD 0, 1, 2, FD 3(pfd[0]), FD 4(pfd[1])   ← оригинал
  Дете:     FD 0, 1, 2, FD 3(pfd[0]), FD 4(pfd[1])   ← копие
```

И двата процеса имат отворен **и двата края** на pipe-а. Трябва да затвориш онзи, който не използваш.

### Типичен pattern: родителят чете, детето пише

```c
int pfd[2];
pipe(pfd);

pid_t pid = fork();
if (pid == 0) {
    // ДЕТЕ: ще пише → затвори read end
    close(pfd[0]);
    // Пренасочи stdout към write end
    dup2(pfd[1], STDOUT_FILENO);
    close(pfd[1]);
    // exec() — наследява новия stdout (= pipe write end)
    execlp("ls", "ls", "-la", NULL);
    err(1, "execlp");
}

// РОДИТЕЛ: ще чете → затвори write end
close(pfd[1]);

// Сега чети — ще получи EOF когато ls приключи
char buf[4096];
ssize_t n;
while ((n = read(pfd[0], buf, sizeof(buf))) > 0) {
    // обработи данните
    write(STDOUT_FILENO, buf, n);
}
close(pfd[0]);
waitpid(pid, NULL, 0);
```

### Pipe между три процеса: `ls | grep | wc`

```
ls ──pfd1──► grep ──pfd2──► wc
```

```c
int pfd1[2], pfd2[2];
pipe(pfd1);
pipe(pfd2);

// Процес ls
if (fork() == 0) {
    close(pfd1[0]); close(pfd2[0]); close(pfd2[1]);
    dup2(pfd1[1], STDOUT_FILENO);
    close(pfd1[1]);
    execlp("ls", "ls", NULL); err(1, "ls");
}

// Процес grep
if (fork() == 0) {
    close(pfd2[0]);
    dup2(pfd1[0], STDIN_FILENO);  close(pfd1[0]);
    dup2(pfd2[1], STDOUT_FILENO); close(pfd1[1]); close(pfd2[1]);
    execlp("grep", "grep", ".c", NULL); err(1, "grep");
}

// Процес wc
if (fork() == 0) {
    close(pfd1[0]); close(pfd1[1]); close(pfd2[1]);
    dup2(pfd2[0], STDIN_FILENO);
    close(pfd2[0]);
    execlp("wc", "wc", "-l", NULL); err(1, "wc");
}

// Родителят затваря всичко и чака
close(pfd1[0]); close(pfd1[1]);
close(pfd2[0]); close(pfd2[1]);
wait(NULL); wait(NULL); wait(NULL);
```

## Named pipes — `mkfifo`

### Unnamed vs Named pipes

**Unnamed pipe** (анонимен) — създава се с `pipe()`, живее само в паметта, достъпен е само за свързани процеси (родител-дете). Изчезва когато всички FD се затворят.

**Named pipe** (FIFO) — има **файл на диска** (специален файл), може да се отвори от **несвързани процеси** само по името му. Данните пак минават само през паметта — файлът на диска е само "адрес", не съдържа реални данни.

### Създаване от C

```c
#include <sys/stat.h>

int mkfifo(const char *path, mode_t mode);
// path → пътят на файла (напр. "/tmp/myfifo")
// mode → права (като chmod: 0644, 0666...)
// Връща: 0 при успех, -1 при грешка (EEXIST ако вече съществува)
```

```c
// Създай named pipe
if (mkfifo("/tmp/myfifo", 0666) == -1) {
    if (errno != EEXIST)   // OK ако вече съществува
        err(1, "mkfifo");
}
```

### Как работи

Named pipe-ът се отваря с обикновен `open()`. Специалното е, че `open()` **блокира** докато и двата края не са отворени:

```
Процес A:                           Процес B:
open("/tmp/myfifo", O_WRONLY)       open("/tmp/myfifo", O_RDONLY)
  → блокира...                        → блокира...
           ↕ двата процеса се "срещат"
  → връща fd                         → връща fd
write(fd, data, len)              →  read(fd, buf, len)
```

```c
// Процес A (писач) — writer.c
int fd = open("/tmp/myfifo", O_WRONLY);  // блокира докато reader не отвори
write(fd, "hello from A", 12);
close(fd);

// Процес B (четец) — reader.c
int fd = open("/tmp/myfifo", O_RDONLY);  // блокира докато writer не отвори
char buf[64];
ssize_t n = read(fd, buf, sizeof(buf));
dprintf(STDOUT_FILENO, "Получих: %.*s\n", (int)n, buf);
close(fd);
```

### Създаване от командния ред

```bash
mkfifo /tmp/myfifo        # създава named pipe
ls -l /tmp/myfifo         # prw-r--r-- ... /tmp/myfifo (p = pipe)

# Терминал 1
cat /tmp/myfifo           # блокира и чака данни

# Терминал 2
echo "hello" > /tmp/myfifo  # изпраща данните — терминал 1 ги показва

rm /tmp/myfifo            # изтрий named pipe
```

## IPC — Inter-Process Communication

**IPC** (Inter-Process Communication) е общото название за всички механизми, чрез които **отделни процеси** обменят данни или се синхронизират. Процесите имат изолирани адресни пространства — не могат директно да четат паметта един на друг, затова имат нужда от специализирани механизми.

### Основните IPC механизми

| Механизъм                | Описание                                            | Кога се ползва                         |
| ------------------------ | --------------------------------------------------- | -------------------------------------- |
| **Unnamed pipe**         | Еднопосочен буфер, само за свързани процеси         | Родител-дете, shell pipelines          |
| **Named pipe (FIFO)**    | Като pipe, но с файлово име — за несвързани процеси | Прости клиент-сървър                   |
| **Signal**               | Асинхронно известие до процес                       | Управление (SIGTERM, SIGCHLD)          |
| **Shared memory**        | Два процеса маппират едни и същи pages              | Голям обем данни, висока скорост       |
| **Message queue**        | Опашка от съобщения в ядрото                        | Асинхронна комуникация                 |
| **Semaphore**            | Брояч за синхронизация                              | Контрол на достъп до споделени ресурси |
| **Socket (Unix domain)** | Двупосочна комуникация по файлов път                | Клиент-сървър на една машина           |
| **Socket (TCP/UDP)**     | Мрежова комуникация                                 | Клиент-сървър между машини             |

### Pipes като IPC

Pipes са най-простият IPC механизъм — вградени в Unix от самото начало. Те са и основата на shell pipeline-ите:

```bash
ls | grep ".c" | wc -l
# ls, grep и wc са три отделни процеса
# комуникират чрез два pipe-а в паметта
```

### Сравнение: pipe vs shared memory

```
Pipe:
  Процес A  →  write(fd, data)  →  [буфер ядро]  →  read(fd, buf)  →  Процес B
  Предимство: прост, автоматична синхронизация
  Недостатък: копиране на данните (user → kernel → user)

Shared memory:
  Процес A                Процес B
  [страница в RAM] ←──────────────→ [страница в RAM]
       ↑                                   ↑
  mmap(MAP_SHARED)              mmap(MAP_SHARED)
  Предимство: без копиране, максимална скорост
  Недостатък: нужна ръчна синхронизация (семафори, mutex)
```

<a id="dup"></a>

## `dup()` и `dup2()`

`dup()` и `dup2()` създават **копие на file descriptor** — нов FD, който сочи към **същото open file entry** в ядрото. Двата FD споделят офсета и са еквивалентни за четене/писане.

### `dup()` — дублирай с най-малкия свободен FD

```c
#include <unistd.h>

int dup(int oldfd);
// Създава копие на oldfd
// Новият FD е най-малкото свободно число
// Връща: новия FD, или -1 при грешка
```

```c
int fd = open("file.txt", O_RDONLY);  // FD 3
int fd2 = dup(fd);                    // FD 4 — копие на FD 3

// fd и fd2 сочат към едно и също open file entry
// Четенето от fd2 придвижва офсета, видим и от fd
read(fd,  buf1, 5);   // чете байтове 0-4, офсет = 5
read(fd2, buf2, 5);   // чете байтове 5-9, офсет = 10 (споделен!)
```

### `dup2()` — дублирай към конкретен FD

```c
int dup2(int oldfd, int newfd);
// Прави newfd да сочи към същото като oldfd
// Ако newfd е отворен — затваря го атомарно
// Връща: newfd, или -1 при грешка
```

```c
int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);  // FD 3

dup2(fd, STDOUT_FILENO);  // FD 1 (stdout) сега сочи към output.txt
close(fd);                // FD 3 вече не е нужен

// Всяко write(STDOUT_FILENO, ...) вече пише в output.txt
write(STDOUT_FILENO, "hello\n", 6);   // → output.txt
```

### Как работи `dup2()` вътрешно

```
Преди dup2(fd, 1):
  FD 1 → [open file entry: stdout/tty]
  FD 3 → [open file entry: output.txt]

dup2(3, 1):
  Затвори FD 1 (stdout)
  FD 1 → [open file entry: output.txt]  ← сочи към output.txt
  FD 3 → [open file entry: output.txt]  ← пак там

close(3):
  FD 3 → (затворен)
  FD 1 → [open file entry: output.txt]  ← само FD 1 сочи
```

### `dup2()` за пренасочване — пълен pattern

```c
// Имплементация на: command > output.txt 2>&1

pid_t pid = fork();
if (pid == 0) {
    // Пренасочи stdout
    int out = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (out == -1) err(1, "open");
    dup2(out, STDOUT_FILENO);   // stdout → файл
    dup2(out, STDERR_FILENO);   // stderr → файл (2>&1)
    close(out);

    execlp("gcc", "gcc", "main.c", NULL);
    err(1, "gcc");
}
waitpid(pid, NULL, 0);
// output.txt съдържа и stdout, и stderr на gcc
```

### Разлика между `dup` и `dup2`

|                       | `dup(fd)`            | `dup2(fd, newfd)`           |
| --------------------- | -------------------- | --------------------------- |
| Нов FD номер          | Най-малкият свободен | Точно `newfd`               |
| Ако `newfd` е отворен | —                    | Затваря го атомарно         |
| Употреба              | Рядко директно       | Стандартен при пренасочване |

## Макроси в C — `#define`

### Какво е макрос

Макросът е инструкция за **препроцесора** — инструментът, който обработва C кода **преди** компилатора. Макросът е буквално **текстова замяна**: препроцесорът търси всяка употреба на макроса и я заменя с дефинирания текст. Компилаторът никога не вижда оригиналните макро имена — вижда само резултата след замяната.

```
Фазите:
  source.c → [Препроцесор] → разширен .c → [Компилатор] → .o → [Линкер] → изпълним
                ↑
          тук живеят макросите (#define, #include, #ifdef...)
```

### Константни макроси

```c
#define PI          3.14159265
#define MAX_SIZE    1024
#define GREETING    "Hello, World!"
#define NEWLINE     '\n'
```

Препроцесорът заменя всяка употреба буквално:

```c
// Написано от теб:
double area = PI * r * r;
char buf[MAX_SIZE];

// Видяно от компилатора (след препроцесиране):
double area = 3.14159265 * r * r;
char buf[1024];
```

```bash
# Виж резултата от препроцесора без компилиране:
gcc -E source.c
```

### Функционални макроси `#define NAME(params) body`

Функционалните макроси изглеждат като функции, но са буквална текстова замяна — не съществуват по време на изпълнение:

```c
#define SQUARE(x)   ((x) * (x))
#define MAX(a, b)   ((a) > (b) ? (a) : (b))
#define MIN(a, b)   ((a) < (b) ? (a) : (b))
#define ABS(x)      ((x) >= 0 ? (x) : -(x))
```

```c
// Написано:
int s = SQUARE(5);
int m = MAX(a, b);

// Видяно от компилатора:
int s = ((5) * (5));
int m = ((a) > (b) ? (a) : (b));
```

### Защо скобите са задължителни

Без скоби текстовата замяна причинява трудни за намиране бъгове:

```c
// ОПАСЕН макрос — без достатъчно скоби:
#define SQUARE_BAD(x)  x * x

SQUARE_BAD(2 + 3)
// Заменя се с: 2 + 3 * 2 + 3  =  2 + 6 + 3  =  11
// Очаквано:    (2+3)*(2+3)     =  25          ← ГРЕШНО!

// ПРАВИЛЕН макрос — всеки параметър и целият израз в скоби:
#define SQUARE(x)  ((x) * (x))

SQUARE(2 + 3)
// Заменя се с: ((2 + 3) * (2 + 3))  =  25   ← правилно
```

**Правилото:** Слагай скоби около **всеки параметър** и около **целия израз** в тялото на макроса.

### Многоредови макроси

```c
#define SWAP(a, b, type) \
    do {                 \
        type _tmp = (a); \
        (a) = (b);       \
        (b) = _tmp;      \
    } while (0)

// Употреба:
int x = 5, y = 10;
SWAP(x, y, int);
// x == 10, y == 5
```

`do { ... } while(0)` е стандартна идиома за многоредови макроси — гарантира, че макросът се държи като единично изречение дори в `if/else`:

```c
// Без do-while — проблем с if/else:
if (condition)
    SWAP_BAD(a, b);  // ← само първият ред е в if-а!
else
    ...

// С do-while — работи правилно:
if (condition)
    SWAP(a, b, int);
else
    ...
```

### `#define` за флагове

В системното програмиране `#define` се използва широко за битови флагове (точно като `O_RDONLY`, `O_CREAT` от `<fcntl.h>`):

```c
// Дефиниране
#define FLAG_READ    (1 << 0)   // 0001 = 1
#define FLAG_WRITE   (1 << 1)   // 0010 = 2
#define FLAG_EXEC    (1 << 2)   // 0100 = 4
#define FLAG_HIDDEN  (1 << 3)   // 1000 = 8

// Употреба
int perms = FLAG_READ | FLAG_WRITE;   // 0011 = 3

if (perms & FLAG_READ)
    dprintf(STDOUT_FILENO, "може да чете\n");
```

### Специални макроси на компилатора

C дефинира няколко вградени макроса:

```c
__FILE__    // "source.c"    — текущият файл
__LINE__    // 42            — текущият ред
__func__    // "main"        — текущата функция
__DATE__    // "Jan  1 2026" — дата на компилация
__TIME__    // "10:30:00"    — час на компилация
```

```c
// Полезни за дебъгване:
#define DEBUG(fmt, ...) \
    dprintf(STDERR_FILENO, "[%s:%d %s] " fmt "\n", \
        __FILE__, __LINE__, __func__, ##__VA_ARGS__)

DEBUG("fd = %d", fd);
// Изход: [main.c:42 main] fd = 3
```

### `#define` vs `const` vs `enum`

```c
// #define — текстова замяна, без тип, без scope
#define MAX 100

// const — истинска C константа, с тип и scope
const int max = 100;

// enum — именувани цели числа, с type checking
enum Color { RED = 0, GREEN = 1, BLUE = 2 };
```

|                    | `#define` | `const`   | `enum`    |
| ------------------ | --------- | --------- | --------- |
| Тип                | няма      | да        | int       |
| Видима за дебъгер  | не        | да        | да        |
| Scope              | файл      | блок/файл | блок/файл |
| Стойност в runtime | не        | да        | не        |

**Препоръка:** За числови константи предпочитай `const` или `enum` пред `#define` — по-безопасно, по-лесно за дебъгване. `#define` е незаменим за условна компилация, флагове и функционални макроси.

### Условна компилация

```c
#define DEBUG_MODE 1

#ifdef DEBUG_MODE
    dprintf(STDERR_FILENO, "debug: x = %d\n", x);
#endif

// Компилиране с/без определение от командния ред:
gcc -DDEBUG_MODE source.c      // дефинира DEBUG_MODE
gcc source.c                   // без дефиниция

// Include guard — предотвратява двойно включване на header:
#ifndef MYHEADER_H
#define MYHEADER_H

// ... съдържанието на header-а ...

#endif  // MYHEADER_H
```

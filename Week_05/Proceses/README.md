# Linux Процеси

## Какво са процесите в Linux?

Всяка програма, която се изпълнява на компютъра, съществува в операционната система като **процес**. Процесът не е просто "изпълняваща се програма" — той е изолирана среда, която съдържа всичко необходимо за изпълнението: код, данни, отворени файлове, мрежови връзки и собствено виртуално адресно пространство в паметта.

### Какво съдържа един процес

Когато ядрото стартира процес, то му отделя собствени ресурси:

- **PID (Process ID)** — уникален идентификатор, число между 1 и ~4 милиона
- **PPID (Parent PID)** — PID на процеса-родител, който го е стартирал
- **Адресно пространство** — виртуална памет: код (text), данни (data), heap, stack
- **File descriptors** — таблица с отворени файлове (stdin=0, stdout=1, stderr=2...)
- **Credentials** — RUID, EUID, RGID, EGID (реален и ефективен потребител/група)
- **Сигнали** — маска от изчакващи и блокирани сигнали
- **Приоритет** — nice стойност (-20 до +19, по-ниска = по-висок приоритет)

### Жизненият цикъл на процеса

Всеки процес в Linux минава през строго дефинирани състояния:

```
fork()                exec()               exit()
  │                     │                    │
  ▼                     ▼                    ▼
CREATED  ──────►  RUNNING  ──────────►  ZOMBIE
                    │   ▲                    │
                    │   │              (изчаква родителя
                 sleep  │               да прочете exit code)
                    │   │                    │
                    ▼   │              wait() от родителя
                 SLEEPING                    │
                                             ▼
                                          REMOVED
```

- **Running (R)** — процесът се изпълнява в момента или чака процесорно време
- **Sleeping (S)** — чака събитие (I/O, таймер, сигнал) — **interruptible** (може да бъде прекъснат)
- **Disk sleep (D)** — чака I/O операция — **uninterruptible** (не може да бъде прекъснат, дори от `kill -9`)
- **Stopped (T)** — спрян с `Ctrl+Z` или сигнал `SIGSTOP`
- **Zombie (Z)** — завършил изпълнение, но родителят не е прочел exit кода (`wait()`)

### Как се раждат процесите — fork и exec

В Linux **всеки процес се ражда от друг процес**. Механизмът е двустъпков:

**`fork()`** — ядрото прави пълно копие на текущия процес. Новият процес (дете) е идентичен с родителя — същия код, същите данни, същите отворени файлове. Единствената разлика е PID-ът.

**`exec()`** — детето извиква `exec()`, което заменя образа му с нова програма. Кодът, данните и стекът се презаписват с новия изпълним файл, но file descriptors и PID остават.

```
bash (PID 1234)
    │
    ├── fork() ──► bash copy (PID 1235)  ← идентичен с родителя
    │                   │
    │               exec("ls") ──► ls (PID 1235)  ← вече е ls, не bash
    │
    └── чака ls да завърши (wait())
```

Това е причината, поради която **PID 1 е специален** — той е `init` (или `systemd` в съвременните дистрибуции) и е родителят на всички останали процеси в системата.

### RUID, EUID и setuid

Всеки процес има два вида идентификатори за потребителя:

**RUID (Real UID)** — реалният потребител, който е стартирал процеса. Не се променя.

**EUID (Effective UID)** — потребителят, чиито **права се прилагат** при проверка на достъп до файлове и ресурси. Обикновено е равен на RUID, но може да се различава при **setuid** програми.

**Setuid** е специален permission бит върху изпълним файл. Когато програма с setuid бит се изпълни, нейният EUID се задава на собственика на файла — не на потребителя, стартирал програмата.

```bash
ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root 68208 Mar 10 /usr/bin/passwd
#     └── s = setuid бит

# Когато ivan изпълни passwd:
# RUID = ivan (реалният потребител)
# EUID = root (защото passwd е собственост на root и има setuid)
# Затова passwd може да пише в /etc/shadow въпреки che ivan не може
```

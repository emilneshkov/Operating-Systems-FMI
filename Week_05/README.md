## `ps` — преглед на процеси

`ps` (**p**rocess **s**tatus) показва моментална снимка на работещите процеси. За разлика от `top`, `ps` не се обновява — изпълнява се веднъж и отпечатва резултата.

### Трите синтаксиса — Unix, BSD и GNU

`ps` поддържа три различни синтаксиса за флагове, наследени от различни Unix традиции. Те **могат да се смесват**, но резултатите понякога се различават:

| Стил                 | Описание                  | Пример          |
| -------------------- | ------------------------- | --------------- |
| **Unix (POSIX)**     | Флаговете започват с `-`  | `ps -ef`        |
| **BSD**              | Флаговете са БЕЗ `-`      | `ps aux`        |
| **GNU long options** | Флаговете започват с `--` | `ps --pid 1234` |

```bash
# Unix стил
ps -e            # всички процеси (-e = every)
ps -f            # пълен формат (full format)
ps -ef           # всички процеси, пълен формат — класическа комбинация
ps -u ivan       # процесите на потребител ivan
ps -p 1234       # конкретен PID

# BSD стил
ps aux           # всички процеси с детайли — най-популярната форма
#  a = процесите на всички потребители с терминал
#  u = user-ориентиран формат (username, CPU, памет)
#  x = включва и процесите БЕЗ терминал (daemon-и)

# GNU long options
ps --pid 1234
ps --ppid 1234   # процесите чиито родител е PID 1234
ps --user ivan
```

### Изходните колони

```bash
ps aux
# USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
# root         1  0.0  0.1 168940 13312 ?        Ss   Mar10   0:03 /sbin/init
# ivan      1234  2.3  1.5 512340 61440 pts/0    Sl   10:22   0:42 /usr/bin/vim
```

| Колона    | Значение                                                  |
| --------- | --------------------------------------------------------- |
| `USER`    | Собственикът на процеса                                   |
| `PID`     | Process ID                                                |
| `%CPU`    | Процент използван CPU (усреднено)                         |
| `%MEM`    | Процент използвана физическа памет                        |
| `VSZ`     | Virtual Size — цялото виртуално адресно пространство (KB) |
| `RSS`     | Resident Set Size — физическата RAM, заета в момента (KB) |
| `TTY`     | Терминалът, свързан с процеса (`?` = без терминал)        |
| `STAT`    | Статус на процеса                                         |
| `START`   | Кога е стартиран                                          |
| `TIME`    | Общо CPU време, консумирано                               |
| `COMMAND` | Командата и аргументите                                   |

### STAT кодове

| Код | Значение                                 |
| --- | ---------------------------------------- |
| `R` | Running — изпълнява се или чака CPU      |
| `S` | Sleeping — interruptible sleep           |
| `D` | Disk sleep — uninterruptible sleep (I/O) |
| `T` | Stopped — спрян с SIGSTOP или Ctrl+Z     |
| `Z` | Zombie — завършил, чака родителя         |
| `s` | Session leader                           |
| `l` | Multi-threaded                           |
| `+` | Foreground process group                 |
| `<` | High priority                            |
| `N` | Low priority (nice)                      |

### `-o` — персонализиран изход

`-o` (output format) позволява да избереш точно кои колони да се показват и в какъв ред:

```bash
ps -eo pid,ppid,user,stat,comm          # избрани колони
ps -eo pid,%cpu,%mem,comm --sort=-%cpu  # сортирано по CPU (низходящо)
ps -eo pid,ppid,user,stat,command       # comm = само името, command = пълен път
```

**Полезни полета за `-o`:**

| Поле      | Значение                    |
| --------- | --------------------------- |
| `pid`     | Process ID                  |
| `ppid`    | Parent PID                  |
| `user`    | Потребителско име (по EUID) |
| `ruser`   | Реален потребител (по RUID) |
| `euid`    | Effective UID (числово)     |
| `ruid`    | Real UID (числово)          |
| `stat`    | Статус                      |
| `%cpu`    | CPU използване              |
| `%mem`    | Памет използване            |
| `vsz`     | Виртуална памет (KB)        |
| `rss`     | Физическа памет (KB)        |
| `comm`    | Само името на командата     |
| `command` | Пълната команда с аргументи |
| `etime`   | Elapsed time — колко работи |
| `lstart`  | Точен час на стартиране     |
| `nice`    | Nice стойност               |

### `ps -o rss` — физическата памет

`rss` (**R**esident **S**et **S**ize) е количеството **физическа RAM**, което процесът използва в момента — за разлика от `vsz`, което включва и виртуална памет, която може никога да не е заредена физически.

```bash
# Покажи процесите сортирани по RSS (най-много памет отгоре)
ps -eo pid,user,rss,comm --sort=-rss | head -15

# RSS в мегабайти (ps дава KB)
ps -eo pid,user,rss,comm --sort=-rss | awk 'NR==1{print} NR>1{$3=int($3/1024)"MB"; print}'

# Общо RSS на всички процеси на потребител ivan
ps -u ivan -o rss | awk 'NR>1{sum+=$1} END{print sum/1024 " MB"}'

# VSZ vs RSS — разликата
ps -eo pid,vsz,rss,comm --sort=-vsz | head -10
# VSZ може да е многократно по-голям от RSS:
# програмата е "резервирала" виртуална памет, но не я е заредила физически
```

### Фамилията от флагове — `ax` и `init` процесът

```bash
ps ax              # всички процеси, включително без терминал
ps axf             # + показва йерархията с ASCII дърво
ps axjf            # + допълнителни колони (session, group)
ps axo stat,euid,ruid,tty,tpgid,sess,pgrp,ppid,pid,pcpu,comm
```

Флагът `x` е ключов — без него `ps` показва само процесите, свързани с терминал. Daemon-ите (nginx, sshd, systemd) нямат терминал и се нуждаят от `x` за да се появят.

```bash
# Намери init процеса — PID 1, родителят на всичко
ps -p 1 -f
# или
ps -eo pid,ppid,comm | grep -E "^  *1 "

# Намери всички директни деца на init
ps -eo pid,ppid,comm | awk '$2==1'

# Намери процес по име
ps aux | grep nginx
ps -C nginx                    # по-чист начин с -C (command)
```

### Практически примери

```bash
# Топ 10 процеса по CPU
ps -eo pid,user,%cpu,comm --sort=-%cpu | head -11

# Топ 10 процеса по памет
ps -eo pid,user,%mem,rss,comm --sort=-%mem | head -11

# Намери zombie процеси
ps aux | awk '$8=="Z"'
ps -eo pid,stat,comm | grep "^[0-9]* Z"

# Всички процеси на конкретен потребител
ps -u ivan -f

# Провери дали процес работи
ps -C nginx > /dev/null && echo "nginx работи" || echo "nginx не работи"

# Покажи пълните команди (без съкращаване)
ps auxww
```

## `pstree` — дърво на процесите

`pstree` показва процесите не като списък, а като **йерархично дърво** — визуализира връзката родител-дете между тях. Изключително полезен за разбиране на структурата на системата.

```bash
pstree                  # дървото на всички процеси
pstree ivan             # само процесите на потребителя ivan
pstree -p              # показва PID-овете
pstree -u              # показва потребителите при промяна
pstree -a              # показва пълните команди с аргументи
pstree -h              # подчертава текущия процес и неговите предци
pstree -n              # сортира по PID вместо по азбука
pstree 1234            # дървото с корен конкретен PID
```

**Примерен изход:**

```
systemd─┬─ModemManager───2*[{ModemManager}]
        ├─NetworkManager─┬─dhclient
        │                └─2*[{NetworkManager}]
        ├─cron
        ├─dbus-daemon
        ├─login───bash───vim
        ├─sshd───sshd───sshd───bash─┬─pstree
        │                           └─htop
        ├─systemd─┬─(sd-pam)
        │  (ivan)  ├─bash───python3
        │          └─tmux: server─┬─bash
        │                         └─bash───nvim
        └─2*[{systemd}]
```

Нотацията `2*[{ModemManager}]` означава два идентични процеса — `pstree` ги групира за четимост.

```bash
# Провери как е стартиран конкретен процес (пълния ancestor chain)
pstree -p -a $(pgrep nginx | head -1)

# Ако pstree не е инсталиран
sudo apt install psmisc        # Debian/Ubuntu
```

`ps axf` дава подобен изход директно без допълнителен инструмент:

```bash
ps axf
# PID TTY      STAT   TIME COMMAND
#   1 ?        Ss     0:03 /sbin/init
# 123 ?        Ss     0:00  \_ sshd
# 456 ?        Ss     0:00  |   \_ sshd
# 457 pts/0    Ss     0:00  |       \_ bash
# 891 pts/0    R+     0:00  |           \_ ps axf
```

## `w` — активни потребители

`w` показва кои потребители са влезли в системата в момента и какво правят. Съчетава информацията от `who` и `uptime` в един изглед.

```bash
w                       # пълен изход
w ivan                  # само активността на потребителя ivan
w -h                    # без header реда
w -s                    # кратък формат (без JCPU и PCPU)
w -f                    # показва от (from) — hostname/IP откъдето е влязъл
```

**Примерен изход:**

```
 10:42:18 up 3 days, 2:15,  3 users,  load average: 0.52, 0.38, 0.31
USER     TTY      FROM             LOGIN@   IDLE JCPU   PCPU WHAT
ivan     pts/0    192.168.1.5      09:15    1:23  0.42s  0.03s vim notes.txt
maria    pts/1    192.168.1.8      10:01    0.00s 0.15s  0.08s bash
root     tty1     -                08:30   2:10m  0.05s  0.05s -bash
```

**Горният ред:**

| Поле                             | Значение                                          |
| -------------------------------- | ------------------------------------------------- |
| Текущ час                        | `10:42:18`                                        |
| `up 3 days, 2:15`                | системата работи непрекъснато от 3 дни и 2 часа   |
| `3 users`                        | трима влезли потребители                          |
| `load average: 0.52, 0.38, 0.31` | средно натоварване за последните 1, 5 и 15 минути |

**Колоните:**

| Колона   | Значение                                                      |
| -------- | ------------------------------------------------------------- |
| `USER`   | Потребителско име                                             |
| `TTY`    | Терминалът (pts/0 = псевдотерминал, tty1 = физическа конзола) |
| `FROM`   | IP адрес или hostname откъдето е влязъл                       |
| `LOGIN@` | Час на влизане                                                |
| `IDLE`   | Колко е неактивен (без вход от клавиатурата)                  |
| `JCPU`   | CPU използване от всички процеси на терминала                 |
| `PCPU`   | CPU използване само от текущата команда                       |
| `WHAT`   | Командата, която изпълнява в момента                          |

**Load average** е средният брой процеси, чакащи или използващи CPU. На система с 1 CPU: `1.0` означава пълно натоварване, `> 1.0` означава опашка. На система с 4 CPU: `4.0` е пълно натоварване.

```bash
# Свързани команди
who                     # по-прост изглед — само кой е влязъл
last                    # история на влизанията/излизанията
last ivan               # само за потребителя ivan
lastb                   # неуспешни опити за влизане
uptime                  # само горния ред от w (uptime + load average)
```

## `diff` — сравнение на файлове

`diff` сравнява два файла ред по ред и показва **какво трябва да се промени**, за да стане първият файл идентичен на втория. Използва се навсякъде — от code review до конфигурационни промени.

```bash
diff file1.txt file2.txt         # основна употреба
diff -u file1.txt file2.txt      # unified формат — по-четим, ползва се в git
diff -i file1.txt file2.txt      # игнорира главни/малки букви
diff -w file1.txt file2.txt      # игнорира whitespace
diff -b file1.txt file2.txt      # игнорира промени в броя интервали
diff -r dir1/ dir2/              # рекурсивно сравнение на директории
diff -q file1.txt file2.txt      # само дали се различават (без детайли)
```

### Как да четеш изхода

**Стандартен формат:**

```
3c3          ← ред 3 от file1 се променя (c = change) в ред 3 на file2
< стар ред   ← < = от file1
---
> нов ред    ← > = от file2

5d4          ← ред 5 от file1 се изтрива (d = delete)
< изтрит ред

7a8          ← след ред 7 на file1 се добавя (a = add) ред 8 от file2
> нов ред
```

Операциите са винаги от гледна точка на **"как да трансформираш file1 в file2"**.

**Unified формат (`-u`) — използва се в `git diff`:**

```diff
--- file1.txt   2024-03-10 10:00:00
+++ file2.txt   2024-03-10 11:00:00
@@ -1,5 +1,5 @@
 непроменен ред
 непроменен ред
-стар ред        ← премахнат от file1
+нов ред         ← добавен в file2
 непроменен ред
```

| Символ         | Значение                                          |
| -------------- | ------------------------------------------------- |
| `-`            | Ред присъства само в **първия** файл (премахнат)  |
| `+`            | Ред присъства само във **втория** файл (добавен)  |
| ` ` (интервал) | Ред е еднакъв и в двата файла (контекст)          |
| `@@`           | Показва номерата на редовете — `@@ -стар +нов @@` |

### Exit кодове

| Код | Значение                             |
| --- | ------------------------------------ |
| `0` | Файловете са **идентични**           |
| `1` | Файловете се **различават**          |
| `2` | Грешка (файлът не съществува и т.н.) |

```bash
# Провери дали два файла са еднакви в скрипт
if diff -q file1.txt file2.txt > /dev/null; then
    echo "Идентични"
else
    echo "Различни"
fi
```

### Практически примери

```bash
# Сравни конфиг преди и след промяна
diff -u nginx.conf.bak nginx.conf

# Рекурсивно сравни два директория
diff -rq dir1/ dir2/

# Покажи само файловете, които се различават (без съдържанието)
diff -rq dir1/ dir2/ | grep "differ"

# Сравни изхода на две команди директно
diff <(ls -la /etc) <(ls -la /etc | sort -k5 -n)

# Сравни отдалечен и локален конфиг
diff <(ssh server cat /etc/nginx/nginx.conf) /etc/nginx/nginx.conf

# Създай patch файл и го приложи
diff -u original.txt modified.txt > changes.patch
patch original.txt < changes.patch
```

### `diff3` — сравнение на три файла

```bash
diff3 mine.txt original.txt theirs.txt   # при merge конфликти
```

## `comm` — общи и уникални редове

`comm` сравнява два **сортирани** файла и показва кои редове са уникални за първия, уникални за втория, и общи за двата. Представи си го като Venn диаграма за файлове.

```bash
comm file1.txt file2.txt     # три колони: само-в-1, само-в-2, общи
```

**Изходът е в три колони:**

```
comm <(echo -e "apple\nbanana\ncherry") <(echo -e "banana\ncherry\ndate")

apple               ← колона 1: само в file1
        date        ← колона 2: само в file2
    banana          ← колона 3: общи за двата
    cherry          ← колона 3: общи за двата
```

### Потискане на колони с `-1`, `-2`, `-3`

Флаговете казват коя колона да **не се показва**:

| Команда          | Показва                                                    |
| ---------------- | ---------------------------------------------------------- |
| `comm -12 f1 f2` | Само **общите** редове (потиска колони 1 и 2)              |
| `comm -13 f1 f2` | Само редовете **само в file2** (потиска колони 1 и 3)      |
| `comm -23 f1 f2` | Само редовете **само в file1** (потиска колони 2 и 3)      |
| `comm -3 f1 f2`  | Уникалните за file1 **и** уникалните за file2 (без общите) |

```bash
# Намери общите редове
comm -12 <(sort file1.txt) <(sort file2.txt)

# Намери редовете само в file1 (липсващи в file2)
comm -23 <(sort file1.txt) <(sort file2.txt)

# Намери редовете само в file2 (нови спрямо file1)
comm -13 <(sort file1.txt) <(sort file2.txt)
```

### `comm` vs `diff` — кога кое да използваш

|                                             | `diff` | `comm`         |
| ------------------------------------------- | ------ | -------------- |
| **Входът трябва да е сортиран**             | Не     | **Да**         |
| **Показва контекст и номера на редове**     | Да     | Не             |
| **Подходящ за множества (set операции)**    | Не     | **Да**         |
| **Показва какво се е променило**            | **Да** | Не             |
| **Лесно извличане на пресечното множество** | Не     | **Да** (`-12`) |

`diff` отговаря на въпроса _"какво се е променило?"_, а `comm` — _"кое е общо и кое е уникално?"_.

### Практически примери

```bash
# Намери пакети, инсталирани на server1 но не на server2
comm -23 <(ssh server1 dpkg -l | awk '{print $2}' | sort) \
         <(ssh server2 dpkg -l | awk '{print $2}' | sort)

# Намери потребители, присъстващи и в двете системи
comm -12 <(cut -d: -f1 /etc/passwd | sort) \
         <(cut -d: -f1 /etc/passwd.bak | sort)

# Намери файлове, изтрити от директория (сравни с backup)
comm -23 <(ls backup/ | sort) <(ls current/ | sort)

# Намери общите IP адреси в два лог файла
comm -12 <(grep -oP '\d+\.\d+\.\d+\.\d+' access.log | sort -u) \
         <(grep -oP '\d+\.\d+\.\d+\.\d+' error.log | sort -u)
```

## Process substitution — `<(command)`

**Process substitution** е функция на bash, която позволява изходът на една команда да се третира като **файл**. Вместо да пишеш резултата в tmp файл и после да го четеш, bash създава именован pipe (`/dev/fd/N`) автоматично.

Синтаксисът е `<(command)` — bash изпълнява `command` в подпроцес и подава пътя до pipe-а вместо истинско файлово име.

```bash
# Без process substitution — нужен е временен файл
sort file1.txt > /tmp/sorted1.txt
sort file2.txt > /tmp/sorted2.txt
diff /tmp/sorted1.txt /tmp/sorted2.txt
rm /tmp/sorted1.txt /tmp/sorted2.txt

# Със process substitution — едно извикване, без временни файлове
diff <(sort file1.txt) <(sort file2.txt)
```

### Как работи вътрешно

Bash форк-ва подпроцес за всяко `<(...)`, свързва изхода му с анонимен pipe и подава пътя `/dev/fd/63` (или `/proc/self/fd/63`) като аргумент на външната команда:

```
diff <(sort file1.txt) <(sort file2.txt)
  │         │                   │
  │    fork → sort file1.txt    fork → sort file2.txt
  │    изход → /dev/fd/63       изход → /dev/fd/62
  │
  └── diff /dev/fd/63 /dev/fd/62
```

```bash
# Виж какво bash реално подава
echo <(echo hello)
# /dev/fd/63        ← това е временен file descriptor, не истински файл
```

### `<(...)` vs `$(...)` — разликата

| Синтаксис    | Какво връща                   | Кога се ползва                                 |
| ------------ | ----------------------------- | ---------------------------------------------- |
| `$(command)` | **Текст** — изходът като низ  | Когато искаш да използваш стойността в команда |
| `<(command)` | **Път до файл** — `/dev/fd/N` | Когато командата очаква файл като аргумент     |

```bash
# $() — изходът се вмъква директно като текст
echo "Днес е $(date)"
lines=$(wc -l < file.txt)

# <() — изходът се подава като файл
diff <(command1) <(command2)
comm <(sort list1.txt) <(sort list2.txt)
wc -l <(find . -name "*.py")
```

### Изходен process substitution — `>(command)`

Съществува и обратната форма `>(command)` — bash пренасочва записа **към** подпроцес вместо към файл:

```bash
# Запиши изхода на две команди едновременно
tee >(gzip > output.gz) >(wc -l) > /dev/null < input.txt
#  └── оригинален файл    └── брои редовете
#                └── записва компресиран файл

# Логвай и на екрана, и във файл
./script.sh > >(tee -a app.log)
```

### Практически примери

```bash

# Провери дали два директории имат еднакви файлове
diff <(ls /dir1 | sort) <(ls /dir2 | sort)

# Намери общи редове в два файла (comm изисква сортирани входове)
comm -12 <(sort file1.txt) <(sort file2.txt)

# Сравни инсталираните пакети на две машини
diff <(ssh server1 dpkg -l) <(ssh server2 dpkg -l)

# Провери дали конфиг файл се е променил спрямо последния commit
diff <(git show HEAD:nginx.conf) nginx.conf

# Подай изхода на команда на програма, която чете само файлове
paste <(cut -d: -f1 /etc/passwd) <(cut -d: -f7 /etc/passwd)

# Изпълни команда върху diff-а на два архива
diff <(tar tf archive1.tar | sort) <(tar tf archive2.tar | sort)
```

### Ограничения

```bash
# НЕ работи с команди, които изискват seekable файл (напр. zip, sqlite)
zip output.zip <(echo "data")   # грешка — zip не може да seek-ва pipe

# НЕ работи в POSIX sh — само в bash, zsh, ksh
sh -c 'cat <(echo hello)'       # грешка в /bin/sh

# Процесът вътре в <() работи асинхронно — внимавай с exit кодове
diff <(command1) <(command2)
echo $?    # exit кодът е на diff, не на command1/command2
```

## Виртуална памет и `vmstat`

### Какво е виртуална памет?

Всеки процес в Linux "вярва", че има достъп до огромно, непрекъснато адресно пространство — на 64-битова система то е теоретично 128 TB. Разбира се, нито един компютър няма толкова RAM. Това е възможно благодарение на **виртуалната памет**.

Ядрото поддържа **page table** за всеки процес — таблица, която превежда виртуалните адреси, които процесът използва, към реалните физически адреси в RAM. Паметта се управлява на блокове от 4KB наречени **pages**.

```
Процес (виртуално)          Физическа RAM
┌──────────────┐            ┌──────────────┐
│ 0x0000_1000  │──────────► │  RAM page 47 │
│ 0x0000_2000  │──────────► │  RAM page 12 │
│ 0x0000_3000  │──────────► │  SWAP page   │ ← не е в RAM, а на диска
│ 0x0000_4000  │──────────► │  RAM page 89 │
└──────────────┘            └──────────────┘
```

### VSZ vs RSS — разликата

**VSZ (Virtual Size)** — цялото виртуално адресно пространство, резервирано от процеса. Включва код, данни, libraries, heap, stack, и паметта, която е **маппирана но не е заредена** физически.

**RSS (Resident Set Size)** — само тези pages, които са **физически в RAM в момента**. Останалите са или на диска (swap), или изобщо не са заредени (ленива инициализация).

```
Процес malloc(100MB):
  VSZ = +100MB  (веднага — паметта е "резервирана" виртуално)
  RSS = +0MB    (засега — ядрото не зарежда физически докато не е нужно)

Процес записва в malloc-натата памет:
  RSS = +100MB  (всеки touched page вече е физически в RAM)
```

Затова VSZ почти винаги е по-голям от RSS — и затова при `ps aux` не трябва да се тревожиш ако VSZ е огромен.

### `vmstat` — статистика за виртуалната памет

`vmstat` дава обобщена статистика за паметта, swap, I/O, CPU и системните извиквания.

```bash
vmstat                  # моментална снимка
vmstat 2                # обновява се на всеки 2 секунди
vmstat 2 10             # 10 обновявания на всеки 2 секунди
vmstat -s               # подробна статистика в списък
vmstat -d               # статистика по дискове
vmstat -m               # slab cache статистика
```

**Примерен изход:**

```
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 1  0      0 524288 102400 819200    0    0    12     8  245  512  5  2 92  1  0
```

**Колоните:**

**procs:**
| | Значение |
|--|----------|
| `r` | Процеси в run queue (чакат CPU) |
| `b` | Процеси в uninterruptible sleep (чакат I/O) |

**memory (KB):**
| | Значение |
|--|----------|
| `swpd` | Памет преместена в swap |
| `free` | Свободна физическа RAM |
| `buff` | Буферна памет (за метаданни на файловата система) |
| `cache` | Кеш памет (съдържание на файлове) |

**swap:**
| | Значение |
|--|----------|
| `si` | Swap in — KB/s, четени от swap в RAM |
| `so` | Swap out — KB/s, изписвани от RAM в swap |

**io:**
| | Значение |
|--|----------|
| `bi` | Blocks in — KB/s, четени от блокови устройства |
| `bo` | Blocks out — KB/s, писани към блокови устройства |

**system:**
| | Значение |
|--|----------|
| `in` | Interrupts за секунда |
| `cs` | Context switches за секунда |

**cpu (%):**
| | Значение |
|--|----------|
| `us` | User space |
| `sy` | System (kernel) |
| `id` | Idle |
| `wa` | I/O wait — CPU чака I/O да завърши |
| `st` | Stolen — отнето от виртуализация |

### Какво да гледаш

```bash
# Следи в реално време
vmstat 1

# Признаци за проблеми:
# so > 0        → системата активно swap-ва — RAM е недостатъчна
# wa > 20%      → CPU чака I/O — бавен диск или прекалено I/O
# b > 0 постоянно → процеси блокирани на I/O
# r > брой_CPU  → CPU е затрупан, процеси чакат в опашка
```
